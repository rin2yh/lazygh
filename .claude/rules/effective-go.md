# 概要

Goコードの実装とレビューで守る Effective Go ベースのルールを記載する。

## フォーマット

- `gofmt` に従う
- 読みやすさを優先し、厳密な行長制限は設けない
- 不要な括弧は書かない
- 短いスコープでは `:=` を優先する

## 命名

- パッケージ名は短く小文字で、可能なら単語1つにする
- `_` や MixedCaps を含むパッケージ名を避ける
- 吃音を避ける
- 小さいスコープの変数名は短く明確にする
- `err`, `ctx`, `i`, `j` など Go で一般的な名前を使う
- インターフェース名に `Interface` サフィックスを付けない
- インターフェース名は `Reader` のように `-er` 系を優先する

## コメント

- すべてのパッケージに package comment を付ける
- export する型、関数、変数にはコメントを付ける
- 処理内容の言い換えではなく、意図や前提を書く

## インターフェース

- インターフェースは小さく保つ
- 実装側ではなく利用側で定義する
- 大きすぎるインターフェースを作らない

## エラー処理

- `error` は最後の戻り値で返す
- 通常の失敗に `panic` を使わない
- error message は小文字始まり、末尾に句読点を付けない
- wrap する場合は失敗文脈が分かる形にする

## 初期化

- `init` は単純な初期化だけに使う
- 引数が必要な初期化は constructor で表現する

## パッケージ設計

- パッケージは単一責務に寄せる
- `utils` のような曖昧なパッケージを作らない
- 依存方向を意識し、循環依存を避ける
- パッケージ名は責務や提供機能が明確に分かる名前にする

## 並行処理

- 並行処理には goroutine を使う
- goroutine 間の連携は共有可変状態より channel を優先する
- channel を close する責務は原則として送信側に置く
- 作成していない channel を受信側で close しない
